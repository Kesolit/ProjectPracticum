# Системный анализ и границы проекта (MVP)

## 1. Границы MVP (Scope)
Согласно заданию, в рамках MVP реализуются следующие возможности:
* **Авторизация**: Регистрация, вход и управление личным профилем.
* **Редактор**: Сборка портфолио из блоков (О себе, Навыки, Проекты, Опыт, Образование, Контакты).
* **Кастомизация**: Выбор визуальной темы (минимум 1–2 варианта оформления).
* **Публикация**: Генерация публичной ссылки и возможность редактирования черновика без поломки публичной версии.

## 2. Пользовательские сценарии (User Stories)
* **Создание**: Как пользователь, я хочу добавлять блоки из списка доступных, чтобы быстро наполнить структуру страницы.
* **Управление**: Как пользователь, я хочу менять порядок блоков (вверх/вниз), чтобы выстроить правильную иерархию информации.
* **Оформление**: Как пользователь, я хочу переключать темы оформления, чтобы подобрать стиль под свои задачи.
* **Публикация**: Как пользователь, я хочу нажать кнопку «Опубликовать», чтобы мое портфолио стало доступно работодателям по ссылке.

## 3. Этапы реализации (Milestones)
* **Показ 1 (3 нед.)**: Защита концепции, архитектуры и схемы данных.
* **Показ 2 (6 нед.)**: Демонстрация ядра (авторизация + каркас редактора).
* **Показ 3 (9 нед.)**: Демонстрация конструктора блоков и превью.
* **Показ 4 (12 нед.)**: Полный цикл от создания до публикации по ссылке.

## 4. Спецификация API для публикации (Sprint 2)

### 4.1 Генерация публичной ссылки (Slug)
* **Эндпоинт**: `PATCH /api/portfolio/{id}/publish`
* **Описание**: Фиксирует текстовый адрес (slug) и переводит портфолио в статус `published`.
* **Запрос**: `{ "slug": "username-link", "isPublic": true }`

### 4.2 Получение публичной версии (Read-Only)
* **Эндпоинт**: `GET /api/public/portfolio/{slug}`
* **Описание**: Открытый маршрут для неавторизованных пользователей. Возвращает данные в формате `IPublicPortfolio`.
* **Особенности**: Бэкенд возвращает секции, заранее отсортированные по полю `order`.

## 5. Логика интеграции микросервисов (C# & Node.js)

### 5.1 Валидация JWT-токенов
Для обеспечения безопасности и единого входа (SSO) используется механизм **Shared Secret**:
1. C# сервер (Auth) подписывает токены с использованием ключа `JWT_SECRET`.
2. Node.js сервер (Drafts) содержит идентичный ключ в переменных окружения.
3. При каждом запросе Node.js сервер самостоятельно проверяет подпись токена без обращения к базе данных C# сервера, извлекая `userId` из полезной нагрузки (payload).

### 5.2 Роутинг и CORS
* Доступ разрешен для локальных адресов разработки (`localhost:5173`, `localhost:3000`).
* Публичные страницы (`/u/*`) обслуживаются Node.js сервером без проверки авторизации.
